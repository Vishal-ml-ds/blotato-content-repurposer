# Content Pillars

> **What Vishal posts about.** Five pillars — every post fits at least one. If a post doesn't fit, it doesn't get published.
> **Status:** DRAFT v0.1 — pillars locked, mix percentages refined after first 20 posts.

---

## The 5 pillars

### 1. Build-in-public (40%)

Real projects shipped, broken, fixed. Code, screenshots, deployment URLs. The journey from beginner to architect, documented.

**Examples:**
- "Built X in 4 hours — here's the FastAPI + Modal setup"
- "Hit a wall with Supabase RLS, here's what fixed it"
- "Shipped my first AI agent — code + lessons"

### 2. Technical breakdowns (25%)

Deep dives on patterns, frameworks, architectures. Code-first explanations of how things work under the hood. Targeting the "I sort of understand this but want to actually understand" reader.

**Examples:**
- "How LangGraph state actually flows — 30 lines of code"
- "RAG with citations: 5-step pipeline, no fluff"
- "Modal vs ECS vs Vercel — when each makes sense"

### 3. Beginner-to-architect journey (15%)

Career-track honest. What's working, what isn't, what changed Vishal's mind. Aimed at people on the same path.

**Examples:**
- "What I learned building 8 AI products in 90 days"
- "The skill that closed the gap between dev and architect"
- "Why I switched from X framework to Y"

### 4. AI architecture deep cuts (15%)

Senior-level patterns: multi-agent orchestration, guardrails, observability, deployment tiers. Aimed at the senior engineers / architects Vishal wants to be peers with.

**Examples:**
- "6-layer guardrail pipeline for production agents"
- "Multi-agent coordination — when to use Nova, when not"
- "Cost analysis: Opus vs Sonnet vs Haiku across 1M tasks"

### 5. Tools / workflow (5%)

Practical setups that compound — Claude Code workflows, prompt patterns, dev environment. Light, useful, low-risk.

**Examples:**
- "My .claude/ setup that cut prompts by 40%"
- "5 Claude Code skills I built and use daily"
- "Why I forked the methodology framework, not just cloned"

## Topics explicitly OFF-LIMITS

- **Day-job work (NDA)** — never reference internal systems, clients, or projects from employer.
- **Other people's projects without explicit permission** — keep their IP separate from my content.
- **Politics, religion, culture wars** — zero benefit, infinite downside.
- **Other people's products without permission** — even praise can backfire.
